## 已修复（本次）

### 1. 【严重】启用 ADMIN_API_TOKEN 后，后台所有写操作 401（鉴权形同虚设）
- **现象**：能打开 `/admin` 登录页、输入令牌也能进后台，但"验证并预览 / 保存监控 / 保存修改 / 删除"全部失败。
- **根因**：登录写的是 httpOnly cookie `sd_token`（值 = 原始令牌），但 `requireWriteAuth` 只认 `Authorization: Bearer`。客户端组件（`MonitorWizard` / `MonitorList`）的 `fetch` 既没带 Bearer（JS 读不到 httpOnly cookie），也没用 cookie —— 于是带 cookie 的请求在写接口被 401 拒回。`start.sh` 默认就生成并启用令牌，所以**刚上线的鉴权实际是"能进后台、不能改任何东西"**。
- **修复**：`src/lib/auth.ts` 的 `requireWriteAuth` 同时接受 `Bearer` 头与 `sd_token` cookie（cookie 为 `sameSite=lax`，跨站请求不携带 → 不引入 CSRF）。新增 `tests/auth.test.ts`（7 例）锁住行为。
- **验证**：`tsc --noEmit` 干净；`pnpm test` 30 全通过。

### 2. 【稳健性】worker 轮询间隔无下限 + 单连接器挂起阻塞全循环
- `WORKER_POLL_INTERVAL_SECONDS=0/负数` → `POLL_INTERVAL_MS=0` → 忙等死循环。
- 某连接器（X API / WeRSS / TrendRadar MCP）长时间不响应时，`runOnce` 整轮被卡住，其他到期任务与下一轮都被拖住。
- **修复**：`src/worker/index.ts` 把轮询间隔钳制到最小 60s；给单次 `gather` 加硬超时（默认 60s，`WORKER_GATHER_TIMEOUT_SECONDS` 可配），超时按 `GATHER_FAILED` 记入该 monitor 的运行记录。
