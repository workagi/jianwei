## 维度核查

### 1. 构建 / 依赖 ✅（1 处待清理）
- `tsc --noEmit` 干净；`eslint` 干净；`pnpm test` 30/30 通过。
- `docker compose config` 合法；`Dockerfile` 的 `tools`/`runner` 多阶段 + `standalone` 正确。
- `pnpm-lock.yaml` 存在，`packageManager` 固定 `pnpm@10.28.1` → 可复现构建。
- ⚠️ **死依赖**：`pg-boss`、`sanitize-html` 在 `src/` 中**从未被引用**（`pg-boss` 是未接入的队列，`sanitize-html` 计划做 XSS 防护但未接线）。不影响运行，但增加体积与供应链面，建议后续清理（或把 sanitize 接上再渲染 `contentHtml`）。

### 2. 安全 / 鉴权 ✅（前一轮已修，2 处运营注意）
- 写接口（POST/PATCH/DELETE `/api/monitors`、POST `/validate`）现接受 `Bearer` 头 **与** 登录 `sd_token` cookie；cookie 为 `sameSite=lax`，跨站不携带 → 无 CSRF。
- 未配置 `ADMIN_API_TOKEN` 时 fail-open（内网/本机模式）；`start.sh` 默认生成随机令牌。
- ⚠️ **fail-open 风险**：若绕过 `start.sh` 直接 `docker compose up` 且未设 `ADMIN_API_TOKEN`，令牌为空 → 全站写操作开放。生产务必经 `start.sh` 或显式设置令牌。
- ⚠️ 登录 cookie 未设 `secure`（HTTP 下正确，但上 HTTPS 后必须加 `secure`）。
- ⚠️ 缺安全响应头（HSTS / X-Frame-Options / CSP）；`/admin` 无 `frame-ancestors` 防护。建议由反向代理或 `next.config` 补。
- ⚠️ `GET /api/monitors`、`GET /api/trendradar/latest` 未鉴权（只读；monitor 配置不含凭据，敏感度低；热榜为公开数据）。如需收紧可给前者加读鉴权、后者加门禁。

### 3. 数据与 DB ✅
- Drizzle ORM 参数化查询；`getItems` 的 ilike 用 sql 模板参数化，无 SQL 注入。
- 迁移齐全（`0000`/`0001` + journal）；`migrate` 服务 `service_completed_successfully` 后才起 web/worker，schema 必定先存在。
- `seed` 幂等（`onConflictDoNothing`/`DoUpdate`），重启安全。
- `monitors.nextRunAt` 有 `defaultNow()` → 新建监控立即被 worker 拾取，无"死监控"。
- `ingest` 用 `(platform, upstreamId)` upsert + `itemMatches` `onConflictDoNothing`，天然幂等，无重复边。

### 4. 部署 / 运维 ⚠️（主要缺口在 HTTPS）
- 仅 `:3000` 对外；postgres/werss/trendradar(-mcp) 均绑 `127.0.0.1`。`.dockerignore`/`.gitignore` 含 `.env*` → 密钥未进镜像/仓库。
- ⚠️ **HTTP 明文**：`:3000` 在 docker 中默认绑 `0.0.0.0`；一旦机器有公网/共享网段 IP，管理员令牌与登录 cookie 以明文传输，可被嗅探。**生产必须前置 HTTPS 反向代理（Caddy/nginx + Let's Encrypt），并给 cookie 加 `secure`。**
- `start.sh` 提供 `doctor`/`status`/`logs` 运维入口；`init_env` 仅在占位/缺失时生成密码与令牌（不破坏既有卷密码一致性）。
- ⚠️ 手动改 `POSTGRES_PASSWORD` 但 postgres 卷已存在时不生效（须删卷重建），`start.sh` 不提示 → 易致全栈停机（见 `MEMORY.md` 部署规则）。
- `web`/`worker`/`trendradar` 无 `healthcheck`（postgres/werss 有）；建议给 web 加 `curl /` 健康检查，便于编排与告警。

### 5. 可靠性 / worker ✅（前一轮已加固，1 处建议）
- 轮询间隔已钳制下限（≥60s），防忙等；单次采集已加硬超时（默认 60s），单连接器挂起不再阻塞整轮。
- X 连接器自带 `AbortSignal.timeout(10_000)`；TrendRadar MCP 解析已修 CRLF。
- 无凭据的连接器构造不崩，错误在 `collect`/`validate` 时抛出并被路由/worker 捕获（→ 502 / 标记 failed），不拖垮进程。
- ⚠️ 失败 monitor 无熔断：错误配置会每 `pollIntervalMinutes` 永久失败，刷 `collection_runs` 与错误日志，`failureCount` 无上限。建议连续失败 N 次后 `enabled=false` 或指数退避。

### 6. 功能完整性 ✅
- 公开首页可正常读库展示信息流；筛选 tab 与搜索框已通过 `?platform=` / `?q=` 接入真实数据库查询，支持按平台与关键词过滤。
- 后台写链路（验证/保存/删除）在启用令牌后已可用（前一轮修复）。
- 采集：trendradar 已内置 seed monitor 自动跑；X/WeChat/Brave 需填真实凭据才有数据（当前 `.env` 为空 → 这几类采集为空，但栈稳定运行）。

### 7. 隐私 / 合规 ✅
- 前端仅渲染纯文本（`bodyText`），**无 `dangerouslySetInnerHTML`** → 当前无 XSS 面。
- `contentHtml` 入库但未渲染；`sanitize-html` 已装未用（如未来渲染 HTML 必须先接线 sanitize）。
- 无 PII 额外落库；monitor 配置不含平台凭据（X/Brave/WeRSS 凭据走环境变量）。
