# 安全策略

## 支持范围

在首个公开稳定版发布前，只维护 `main` 分支的最新提交。正式发布后，安全修复将优先覆盖最新版本；旧版本可能需要先升级。

## 私下报告漏洞

请不要为以下问题创建公开 Issue：

- API Key、Cookie、会话或用户内容泄漏。
- 未授权访问管理后台或写 API。
- 凭据加密、密码哈希或权限边界失效。
- 能导致任意代码执行、服务端请求伪造或持久化脚本执行的问题。
- 会扩大第三方账号风险或绕过平台安全限制的行为。

首选方式是在仓库的 **Security → Report a vulnerability** 中使用 GitHub 私有漏洞报告。若该入口尚未启用，请先通过维护者 GitHub 主页请求一个私下沟通渠道，不要在请求中附上可利用细节或真实凭据。

一份有效报告应包含：

- 受影响版本或提交 SHA。
- 部署方式和最小复现步骤。
- 实际影响与攻击前提。
- 已进行的脱敏处理。
- 如果知道，建议的修复方向。

维护者会尽快确认收到报告，并在能够安全公开后记录修复。当前项目不提供漏洞赏金计划。

## 安全边界

- 见微面向单用户自托管，不是多租户 SaaS 权限系统。
- 阅读页可以公开；管理页和写 API 必须受保护。
- WeRSS、TrendRadar、wechat-download-api 等 Sidecar 默认不应直接暴露到公网。
- 第三方来源返回的内容均视为不可信输入，不能当作代码或管理指令执行。
- 平台登录、额度、Cookie 过期和风控限制本身不属于见微安全漏洞。

生产部署请同时阅读 [生产部署说明](docs/production-deploy.md)。
